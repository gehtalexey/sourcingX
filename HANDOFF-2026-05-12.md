## Pending discussions to pick up tomorrow

### A. PR #43 performance: post-enrichment → Filter+ takes 1-2 min, target a few seconds

Plain-English summary of the bottleneck:

The "Enrichment complete!" message shows the moment Crustdata returns. After that, ~6 hidden steps run before Filter+ becomes clickable. Most of them aren't actually required for Filter+ to work — Filter+ only needs the in-memory profile list, which is filled in well before the rest finishes.

**Time budget for 500 profiles:**
- `flatten_for_csv` (reshape Crustdata response): 5-20s
- Email/name preservation loop (slow pandas iterrows + .at): 1-3s
- Merge + dedupe with existing enriched: < 1s
- Save session to disk (JSON): < 1s
- **Save 500 profiles back to Supabase in batches of 100: 15-30s** ← biggest blocker
- `st.cache_data.clear()` wipes ALL caches globally: < 1s
- Full Streamlit rerun: 2-5s

For 1000+ profiles, this stacks to 60-120s.

**Three proposed changes, ordered by impact:**

1. **Move the Supabase save to a background thread** so the UI isn't blocked. Show a small "💾 Saving to database..." indicator that doesn't gate Filter+. Saves 15-30s.
2. **Stop the global cache wipe.** Today the code calls `st.cache_data.clear()` which nukes search cache, profile pool cache, and the new PR #43 cache. Replace with surgical clears (`_cached_recently_enriched_urls`, the new auto-load cache, etc.). Saves 1-3s on next Filter+ visit.
3. **Vectorize the preservation loop** — replace `iterrows() + .at` with a `merge` on linkedin_url. 5-10× faster. Saves 1-3s.

**Combined target:** ~3-5 seconds from "Enrichment complete" → Filter+ usable.

**Risks worth weighing:**
- Background save means if the user closes the tab mid-save, a small window of unpersisted data exists.
- A failed background save is silent unless we surface it (indicator should turn red).
- Surgical cache clear needs testing — wrong list = stale "already enriched" counts.

**Open question:** ship all three as one PR or just #1 first?

### C. PR #46 Codex blocker — the proper fix

The column-only path (`search_profiles_boolean` in db.py) needs to OR the title query across `current_title` AND `raw_data->>'headline'` server-side.

**Confirmed working at SQL level** (tested via Supabase MCP):

```sql
SELECT count(*) FROM profiles
WHERE current_title ILIKE '%python engineer%'
   OR raw_data->>'headline' ILIKE '%python engineer%';
-- returns 5 (proof of concept)
```

**The PostgREST translation** needs a multi-column boolean AST converter. Two design options I considered:

1. **Leaf-level OR via a new `_ast_to_postgrest_multi(ast, columns)` helper.** Each TERM becomes `or(col1.ilike.*val*, col2.ilike.*val*)`. NOT TERM becomes `and(col1.not.ilike.*val*, col2.not.ilike.*val*)` (De Morgan). Semantically correct for nested AND/OR.
2. **Two parallel PostgREST queries**, one per column, then union results client-side. Simpler PostgREST syntax (no jsonb-in-or() unknowns) but doubles round-trips and complicates pagination.

I was leaning toward option 1. Should pick it up by:
- Adding the new helper in db.py near `_ast_to_postgrest`
- Adding `parse_boolean_query_multi(query, columns)` public wrapper
- In `search_profiles_boolean`, pass `['current_title', 'raw_data->>headline']` for the `current_title` filter
- Add a regression test specifically for the server-side / column-only path (Codex called this out)
